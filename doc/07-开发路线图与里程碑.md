# 07 开发路线图与里程碑

## 1. 计划周期

建议以 12~16 周为一个完整二开周期，分 4 个阶段推进。

## 2. 里程碑总览

| 阶段 | 周期 | 目标 | 关键交付 |
| --- | --- | --- | --- |
| M1 基线修复 | 第 1~2 周 | 跑通工程基线、统一环境 | lint/typecheck/unit 稳定 |
| M2 架构分层 | 第 3~6 周 | 抽象接口、收敛模块边界 | Runtime/Style/Data 契约 |
| M3 功能扩展 | 第 7~11 周 | 引入二开能力并验证 | 扩展 SDK + 业务能力 |
| M4 性能与发布 | 第 12~16 周 | 性能达标、兼容发布 | 性能报告 + 灰度上线 |

## 3. 阶段详情

## 3.1 M1 基线修复（第 1~2 周）

- 对齐 Node/npm 版本与本地脚本；
- 修复 lint 与 unit 运行问题；
- 建立“每日质量检查”流水线。

DoD：

- 核心开发环境一键启动；
- 关键质量门禁稳定通过；
- 问题台账建立并可追踪。

## 3.2 M2 架构分层（第 3~6 周）

- 完成 Runtime Core 契约抽离；
- 完成 Style/Data 交互契约；
- 建立模块依赖约束规则。

DoD：

- 新增代码符合模块边界；
- 核心场景无功能回归；
- 架构文档更新完成。

## 3.3 M3 功能扩展（第 7~11 周）

- 实现扩展 SDK 的统一入口；
- 沉淀 overlay/source/plugin 的标准接入模板；
- 与业务场景进行联调验证。

DoD：

- 至少 1~2 个真实业务扩展示例落地；
- 接口文档 + 示例代码齐备；
- 关键扩展链路具备自动化回归。

## 3.4 M4 性能与发布（第 12~16 周）

- 完成性能专项优化并输出对比报告；
- 执行兼容性回归与灰度发布；
- 形成最终复盘报告。

DoD：

- 指标达到预设门槛；
- 无阻断级兼容回归；
- 发布流程可复用。

## 4. 资源配置建议

## 4.1 5 人团队参考

- 架构/平台：1 人
- 引擎/渲染：2 人
- 数据/扩展：1 人
- 测试/发布：1 人（可兼职）

## 4.2 角色职责

- 技术负责人：里程碑验收与风险决策
- 模块 Owner：方案落地与质量兜底
- QA/测试：回归体系与发布检查

## 5. 节奏管理

- 周节奏：周计划 -> 日推进 -> 周复盘；
- 每周固定 1 次架构/接口评审；
- 每个里程碑结束做 Go/No-Go 评审。

## 6. 里程碑风险闸门

- 若 M1 未完成，不进入 M2；
- 若 M2 模块边界未稳定，不大规模扩展功能；
- 若 M4 指标不达标，不全量发布。
